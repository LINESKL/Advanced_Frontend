## 1. Информация об использовании AI
- **Инструмент:** Google Gemini
- **Дата:** 2026-02-28

## 2. Конкретные промпты

### Промпт 1:
"Почему Error Boundary в React должен быть только классовым компонентом? Можно ли реализовать его через функциональный компонент?"

**Что получил:** Разъяснение, что методы жизненного цикла `getDerivedStateFromError` и `componentDidCatch` не имеют аналогов в хуках, поэтому использование классов обязательно для этой задачи.

### Промпт 2:
"Как реализовать функцию сброса ошибки (Retry) в Error Boundary, чтобы пользователь мог попробовать загрузить компонент снова без перезагрузки всей страницы?"

**Что получил:** Логику метода `resetError`, который меняет состояние `hasError` обратно на `false`, и пример вызова этой функции из дочернего компонента.

### Промпт 3:
"Как типизировать Error Boundary на TypeScript, чтобы избежать использования 'any' в Props и State?"

**Что получил:** Шаблон интерфейсов для `Props` (включая `ReactNode`) и `State`, обеспечивающий строгую проверку типов в классовом компоненте.

## 3. Модификация и проверка
- **Типизация:** Описал строгие интерфейсы для пропсов, включая передачу кастомного компонента `fallback`.
- **Иерархия компонентов:** Самостоятельно определил правильное место для `ErrorBoundary` в дереве компонентов (снаружи `Suspense`), чтобы перехватывать ошибки загрузки чанков.
- **Тестирование:** Симулировал ошибки сети через "Offline" режим в браузере и намеренно ломал пути импорта, чтобы проверить срабатывание запасного UI.

## 4. Что узнал
- Изучил специфику работы предохранителей (Error Boundaries) в React и их ограничения (не ловят ошибки в обработчиках событий и асинхронном коде).
- Понял важность предоставления пользователю возможности восстановить работу приложения (Retry) вместо показа пустого экрана.
